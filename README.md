# Online Store Backend (Spring Data JPA) 🛒

Educational backend project for an online store, developed with Spring Boot and Spring Data JPA.
Ein cloudbasiertes Lernprojekt für ein Online-Shop-Backend, entwickelt mit Spring Boot und Spring Data JPA.

---

## 🇺🇸 English Version

### 📦 Core Features
* **Product Management:** Save, update, delete, and retrieve products.
* **Customer Management:** Handle customer data and statuses.
* **Cart Logic:** Add/remove items, clear cart, calculate total cost and average price.

### 🛠️ Tech Stack
* **Java 17+**
* **Spring Boot (Data JPA, Web)**
* **Hibernate / PostgreSQL / H2**
* **MapStruct** — compilation-time code generation for Entity-to-DTO conversion
* **Lombok** & **Maven**

### 🧩 Architecture & Project Structure
The project follows a solid Layered Architecture approach:
1. **Entity Layer (Domain):** Standard JPA entities representing the database schema (`Customer`, `Product`, `Position`, `Cart`).
2. **DTO Layer (Data Transfer Object):** Decoupled data structures for safe API contract fulfillment (`ProductDto`, `PositionDto`, `CustomerDto`).
3. **Mapping Layer:** Abstract interfaces where MapStruct automatically generates the underlying active implementations during the compilation phase.
4. **Repository Layer:** Spring Data JPA interfaces for database abstractions and operations.
5. **Service Layer:** Core business logic components (`CustomerServiceImpl`, `ProductServiceImpl`).

### 💡 MapStruct Technical Insights & `mvn clean compile`
**MapStruct** is a compile-time annotation processor. Unlike reflection-based mappers, it generates plain, fast Java code at compile time. When you write an abstract interface like `PositionMapper`, you only declare what needs to be mapped.

When executing the terminal command:
```bash
mvn clean compile

```

The Maven lifecycle performs two essential phases:

1. **`clean`** — Completely deletes the `target/` output folder to prevent conflicts from old class files and guarantee a fresh build environment.
2. **`compile`** — Triggers the standard Java Compiler along with the MapStruct processor. MapStruct reads the interfaces and autogenerated concrete Java classes (e.g., `PositionMapperImpl.java`) inside `target/generated-sources/annotations/`.

#### Nested Mapping Dependencies via `uses`

When a specific mapper requires handling a nested entity relationship (such as converting a nested `Product` inside a `Position`), MapStruct is instructed to wire in another mapper using the `uses` attribute:

```java
@Mapper(componentModel = "spring", uses = {ProductMapper.class})
public interface PositionMapper { ... }

```

During compilation, MapStruct generates precise code invoking the dependent mapper:

```java
positionDto.setProduct( productMapper.mapEntityToDto( entity.getProduct() ) );

```

> ⚠️ **Critical Notice:** As highlighted by the yellow IDE warning strip in IntelliJ IDEA, files within the `target/generated-sources/` folder must **never** be manually modified. Any local edits will be permanently wiped out during the next build or compilation cycle.

### 🧩 Architektur & Projektstruktur

Das Projekt folgt einer klassischen Schichtenarchitektur (Layered Architecture):

1. **Entity-Schicht (Domain):** Standard-JPA-Entitäten, die das Datenbankschema repräsentieren (`Customer`, `Product`, `Position`, `Cart`).
2. **DTO-Schicht (Data Transfer Object):** Entkoppelte Datenstrukturen für eine sichere API-Kommunikation (`ProductDto`, `PositionDto`, `CustomerDto`).
3. **Mapping-Schicht:** Abstrakte Schnittstellen (Interfaces), für die MapStruct während der Kompilierungsphase automatisch die konkreten Implementierungen generiert.
4. **Repository-Schicht:** Spring Data JPA-Schnittstellen für Datenbankabstraktionen und -operationen.
5. **Service-Schicht:** Komponenten für die zentrale Geschäftslogik (`CustomerServiceImpl`, `ProductServiceImpl`).

### 💡 MapStruct Technische Einblicke & `mvn clean compile`

**MapStruct** ist ein Annotation-Prozessor, der zur Kompilierungszeit arbeitet. Im Gegensatz zu reflexionsbasierten Mappen generiert er schnellen Java-Code zur Compile-Zeit. Wenn Sie ein abstraktes Interface wie `PositionMapper` schreiben, deklarieren Sie nur, was gemappt werden muss.

Bei der Ausführung des Terminal-Befehls:

```bash
mvn clean compile

```

führt der Maven-Lebenszyklus zwei wichtige Phasen aus:

1. **`clean`** — Löscht den gesamten `target/`-Ausgabeordner vollständig, um Konflikte mit alten Klassendateien zu vermeiden und eine saubere Build-Umgebung zu garantieren.
2. **`compile`** — Aktiviert den Java-Compiler zusammen mit dem MapStruct-Prozessor. MapStruct liest die Interfaces und generiert automatisch die konkreten Java-Klassen (z. B. `PositionMapperImpl.java`) im Ordner `target/generated-sources/annotations/`.

#### Verschachtelte Mapping-Abhängigkeiten via `uses`

Wenn ein bestimmter Mapper eine verschachtelte Entitätsbeziehung verarbeiten muss (z. B. das Konvertieren eines verschachtelten `Product` innerhalb einer `Position`), wird MapStruct angewiesen, einen anderen Mapper über das Attribut `uses` einzubinden:

```java
@Mapper(componentModel = "spring", uses = {ProductMapper.class})
public interface PositionMapper { ... }

```

Während der Kompilierung generiert MapStruct präzisen Code, der den abhängigen Mapper aufruft:

```java
positionDto.setProduct( productMapper.mapEntityToDto( entity.getProduct() ) );

```

> ⚠️ **Wichtiger Hinweis:** Wie durch den gelben IDE-Warnbalken in IntelliJ IDEA hervorgehoben, dürfen Dateien im Ordner `target/generated-sources/` **niemals** manuell geändert werden. Alle lokalen Änderungen werden beim nächsten Build- oder Kompilierungszyklus vollständig überschrieben und gelöscht.
