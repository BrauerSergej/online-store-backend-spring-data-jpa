Вот подробная таблица всех основных сценариев возникновения исключений (Exceptions) для твоего интернет-магазина.

Я разбил их на логические группы (уровни), чтобы тебе было легко ориентироваться, когда и какую ошибку использовать.

### 🗄️ Таблица сценариев использования Exceptions в REST API

| Сценарий (Что произошло?) | На каком уровне? | Какой Exception выбрасывается? | HTTP Статус для клиента | Суть ответа (Что видит клиент) |
| --- | --- | --- | --- | --- |
| **1. Ошибки формата (Валидация)** |  |  |  |  |
| Пользователь прислал пустое имя или отрицательную цену. | DTO (Аннотации @NotNull, @Min) | ConstraintViolationException (или MethodArgumentNotValidException) | **400 Bad Request** | «Исправьте ошибки в форме (имя не может быть пустым)». |
| Пользователь прислал буквы вместо цифр (например, в поле ID). | Контроллер / Jackson (Парсер) | HttpMessageNotReadableException или TypeMismatchException | **400 Bad Request** | «Неверный формат данных». |
| **2. Ошибки поиска в базе данных** |  |  |  |  |
| Запросили товар по ID = 999, а его нет в базе. | Сервис (orElseThrow) | EntityNotFoundException (Твой кастомный класс) | **404 Not Found** | «Товар с ID 999 не найден». |
| Пытаются удалить или обновить юзера, которого не существует. | Сервис (orElseThrow) | EntityNotFoundException | **404 Not Found** | «Пользователь не найден». |
| **3. Нарушение бизнес-правил (Ошибки логики)** |  |  |  |  |
| Пытаются посчитать стоимость пустой корзины. | Сервис (Твой if) | CartEmptyException (Твой кастомный класс) | **400 Bad Request** | «Корзина пуста, расчет невозможен». |
| Пытаются удалить из корзины товар, которого там никогда не было. | Сервис (Твой if) | IllegalArgumentException (или кастомный ProductNotInCartException) | **400 Bad Request** | «Данного товара нет в вашей корзине». |
| Пытаются добавить в корзину товар, который снят с продажи (active=false). | Сервис (Твой if) | ProductNotActiveException (Твой кастомный класс) | **400 Bad Request** | «Этот товар больше не продается». |
| Регистрация нового покупателя с email, который уже есть в базе. | Сервис (Твой if) | DuplicateEmailException (Твой кастомный класс) | **409 Conflict** | «Пользователь с таким email уже существует». |
| Попытка добавить в корзину 10 товаров, а на складе осталось только 2. | Сервис (Твой if) | ProductOutOfStockException (Твой кастомный класс) | **409 Conflict** | «Недостаточно товара на складе». |
| **4. Системные сбои (Баги программиста)** |  |  |  |  |
| Код попытался обратиться к пустому объекту (забыли проверку на null). | В любом месте кода | NullPointerException (Встроенный в Java) | **500 Internal Server Error** | «Внутренняя ошибка сервера. Попробуйте позже». |
| Упала база данных, или неправильно написан SQL-запрос. | Репозиторий (Hibernate) | SQLException или DataAccessException | **500 Internal Server Error** | «Внутренняя ошибка сервера. Попробуйте позже». |

---

### Как пользоваться этой таблицей на практике:

1. **Если ты пишешь проверку в DTO** — тебе вообще не нужно думать об Exceptions. Просто ставь аннотации (@NotNull, @NotBlank), Spring сам кинет ConstraintViolationException (из группы 1).
2. **Если ты пишешь метод findById** — всегда завершай его .orElseThrow(() -> new EntityNotFoundException(...)) (из группы 2).
3. **Если ты пишешь любой другой if в Сервисе** — смотри в группу 3. Подумай, это конфликт данных (тогда статус 409 и создаешь свой Exception) или глупый запрос клиента (тогда статус 400 и кидаешь кастомный Exception или стандартный IllegalArgumentException).
4. **Ошибки из группы 4 ты специально никогда не вызываешь (throw)!** Они вылетают сами, когда ты допускаешь баг в коде. Твоя задача — просто поймать их в GlobalExceptionHandler, чтобы скрыть от пользователя страшный технический текст.
