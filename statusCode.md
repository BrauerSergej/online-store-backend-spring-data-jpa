Прекрасная новость: **их не нужно запоминать все!**

Даже опытные разработчики не помнят наизусть все 60+ существующих HTTP-статусов. Для написания отличного REST API тебе понадобится знать всего около 6 штук.

Чтобы легко ориентироваться, достаточно запомнить «правило первой цифры». Первая цифра статуса сразу говорит о том, **кто виноват** или **что происходит**.

## Глобальная логика (Правило первой цифры)

* **1xx (Информационные):** «Подожди, процесс идет». В повседневной разработке мы их почти не трогаем руками.
* **2xx (Успех):** «Всё отлично!». Запрос прошел, данные сохранены или выданы (например, 200 OK или 201 Created).
* **3xx (Перенаправление):** «Тебе в другую дверь». Ресурс переехал, иди по другому адресу.
* **4xx (Ошибка клиента):** **Виноват клиент.** Пользователь прислал кривые данные, забыл пароль, лезет куда нельзя. Сервер работает нормально, он просто отказывается выполнять глупый запрос.
* **5xx (Ошибка сервера):** **Виноват программист.** Клиент всё сделал правильно, но твой код сломался (например, вылетел NullPointerException или отвалилась база данных).

## 6 главных статусов для обработки ошибок

Это твоя шпаргалка. Если у тебя в приложении произошла ошибка, просто выбери наиболее подходящую ситуацию из таблицы:

| Статус | Название | Ситуация (Когда использовать) |
| --- | --- | --- |
| **400** | Bad Request | Пользователь прислал неверные данные. Например: пустое имя, отрицательный возраст, неверный формат email. |
| **401** | Unauthorized | Проблема с узнаванием. Пользователь не прислал токен или ввел неверный логин/пароль. Ответ на вопрос: "Ты кто такой?". |
| **403** | Forbidden | Проблема с правами. Мы знаем, кто этот юзер, но ему сюда нельзя. Например: обычный покупатель пытается зайти в админ-панель. |
| **404** | Not Found | Того, что просят, не существует. Неверный URL в браузере или поиск товара с ID, которого нет в базе данных. |
| **409** | Conflict | Конфликт бизнес-логики. Данные правильные, но их нельзя применить. Например: попытка зарегистрироваться с email, который уже есть в базе. |
| **500** | Internal Server Error | Критическая поломка кода. Сервер не смог обработать запрос из-за бага разработчика. |

## Простой алгоритм для выбора (Чит-шит)

Когда пишешь код и думаешь, какой статус прицепить к ошибке, задай себе три вопроса:

1. **Код упал с технической ошибкой?**
   Да ➡️ 500 (клиенту отдаем безликое сообщение, в консоль пишем весь лог).
2. **Запрос вообще можно выполнить с такими данными?**
   Нет, данные кривые ➡️ 400.
3. **Данные идеальные, но нарушают логику приложения?**
* Объекта просто нет ➡️ 404.
* Такой объект уже есть (дубликат) ➡️ 409.



Чаще всего в GlobalExceptionHandler у тебя будет именно такой набор: метод для 404 (объект не найден), метод для 400 (ошибки валидации), метод для 409 (дубликаты) и один метод-ловушка для 500 (непредвиденные поломки кода).
